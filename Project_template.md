# Анализ требований

## 1. Ограничения
* Документация хранится в разных источниках (Confluence, Notion, PDF).
* Есть дубликаты документов, т.е. необходим контроль версий.
* В документации может быть чувствительная для бизнеса информация. Нельзя допустить утечки в облако такой информации в открытом виде.

## 2. Требования
* Персонализированные и точные ответы.
* Работа с конфиденциальными документами.
* Сокращение FTE-затрат на поиск информации.
* Обнаружение пробелов в документации
* Масштабируемость, расширение на другие команды.

## 3. Сравнение LLM-моделей

| Критерий | Локальные LLM (Hugging Face, Llama 3 / Qwen)                                                | Облачные LLM (OpenAI, YandexGPT)                                                                |
|----------|---------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------|
| Качество ответов | При хороших настройках промпта качесто будет высоким, но будет уступать облачным вариантам. | Отличное, лучше понимают контекст чем локальные варинты, лучше справляются с большими текстами. |
| Скорость | Зависит от конфигурации локального сервера.                                                 | Очень высокая, SLA от провайдера                                                                |
| Стоимость владения | Высокие первоначальные затраты, сервера и тп.                                               | Стабильные затраты постоянные затраты на токены, есть возможность оптимизироваться.             |
| Удобство развёртывания | Требуется развертывание и настройка, возможно потребуется ML инженер.                       | Очень просто интегрируется.                                                                     |
| Безопасность | Чувствительная информация не покидает локальной сети компании.                              | Для защиты данных потребуется дополнительная настройка, в том числе шифрование данных.          |
| Масштабируемость | Сложнее масштабировать                                                                      | Очень просто масштабировать                                                                     |

## 4. Сравнение моделей эмбеддингов

| Критерий | Локальные Sentence-Transformers                      | Облачные OpenAI Embeddings                                         |
|----------|------------------------------------------------------|--------------------------------------------------------------------|
| Скорость создания индекса | Достаточно бытсрая, но уступает облачным             | Очень высокая, масштабируемая                                      |
| Качество поиска | Хорошее качество поиска, уступает облачным           | Отличное, особенно для длинных и сложных документов                |
| Стоимость владения | Бесплатно.                                           | Оплата за токены, может быть ощутимо при большом объёме документов |
| Конфиденциальность | Данные не покидают локальную сеть компании           | Данные уходят в облако, требуется как минимум шифрование           |
| Масштабируемость | Возможность есть, но завист от конфигурации серверов | Очень простая, масштабируется через API                            |

## 5. Сравнение векторных БД

| Критерий | FAISS | ChromaDB |
|----------|-------|-----------|
| Скорость поиска и индексации | Очень высокая, подходит для больших индексов | Высокая, достаточная для десятков/сотен тысяч документов |
| Сложность внедрения | Средняя; требует написания слоя для метаданных и API | Низкая; готовый  API, поддержка метаданных и фильтров |
| Удобство работы | Низкое; нужно реализовать персистентность и фильтрацию вручную | Высокое; встроенная персистентность и фильтрация «из коробки» |
| Стоимость владения | Бесплатно; нужно только инфраструктура и разработка | Бесплатно; инфраструктура примерно та же, но меньше DevOps-нагрузки |
| Масштабируемость | Очень высокая при правильной реализации | Высокая, но чуть ниже FAISS на миллионах документов |

## 5. Предлагаемая конфигурация сервера
Исходя из объемов задания:
* ≈ 18 000 Markdown/MDX файлов в репозиториях.
* ≈ 3 000 страниц Confluence (30 пространств).
* ≈ 250 PDF‑спецификаций (наследие заказных проектов).
* Объём прироста — примерно 400 страниц в месяц.

Конфигурация: 32vCPU, 128GB RAM, 1×GPU (24GB).

## 6. Варианты реализации

| Вариант | LLM | Embeddings | Векторная база      | Плюсы                                                                                                                                           | Минусы                                                                             |
|---------|-----|------------|---------------------|-------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------|
| A. Полностью облачный | OpenAI | OpenAI | ChromaDB (облако)   | Высокое качество ответов, быстрая интеграция                                                                                                    | Риски безопасности, регулярные расходы выше, чувствительные данные уходят в облако |
| B. Гибридный | OpenAI | bge-large (локально) | ChromaDB (локально) | Высокое качество ответов, контроль конфиденциальных данных, хранение метаданных и версий, регулярные расходы все равно остаются (токены OpenAI) | Нужно поддерживать сервер и GPU, DevOps                                            |
| C. Полностью локальный | Llama 3 / Qwen | Sentence-Transformers / bge-large | ChromaDB (локально) | Максимальная безопасность, нет оплаты за токены                                                                                                 | Высокий затраты, сложная поддержка, ниже качество генерации, требуется ML-команда  |

Лучшим выбор для компании будет являться гибридный вариант.
Аргументы:
1. Чувствительные данные защищены, находятся в локальной сети.
2. Легкая масштабируемость и нет больших начальных затрат.
3. Не требуется найм ML-специалистов.
